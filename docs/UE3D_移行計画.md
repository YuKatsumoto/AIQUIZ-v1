# AI脱出クイズ — Unreal Engine 3D 化 移行計画

## 1. 目的とスコープ

- **目的**: 現在の Pygame 2D「AI脱出クイズ」を Unreal Engine で 3D 化し、同じルール・出題システムで遊べるようにする。
- **スコープ**: 単体プレイをまず完成させ、2人プレイ・履歴・評価は後段で対応する方針も可。

---

## 2. 現行ゲームの整理（再確認）

| 項目 | 内容 |
|------|------|
| コアループ | 問題表示 → 壁が迫る → プレイヤーが左右に移動 → 正解/不正解のドアに進入で判定 |
| 問題形式 | 二択（選択肢2つ）。問題文 + 左ドア = 選択肢1、右ドア = 選択肢2 |
| 出題元 | ONLINE: OpenAI + Gemini で生成 / OFFLINE: `offline_bank.json` + 内蔵画像問題 |
| プレイモード | 10問チャレンジ / エンドレス |
| 設定 | 学年(1–6)、教科(算数/理科/国語)、難易度(簡単/普通/難しい)、速度、1人/2人 |
| その他 | 診断モード、問題評価(いいね/いまいち)、履歴、Firebase 連携 |

---

## 3. 全体アーキテクチャ

### 3.1 推奨: 「クイズ API サーバー + UE クライアント」

- **Python クイズサーバー（新規）**
  - 既存の `2D_pygame.py` から「問題取得・生成・オフラインバンク・評価」まわりを **REST API 化**して切り出す。
  - 役割: 問題の取得/生成、オフライン問題の配信、評価の記録。API キーはサーバー側のみ保持。
- **Unreal Engine クライアント**
  - HTTP で上記 API を叩き、問題 JSON を受け取る。描画・入力・ゲームルールはすべて UE で実装。

**メリット**
- OpenAI/Gemini の呼び出し・プロンプト・評価ロジックを Python のまま再利用できる。
- API キーを UE や配布物に含めなくてよい。
- 同じ API を Pygame 版と UE 版の両方で使える（将来の Web 版も可）。

**代替案**
- **オフライン専用**: UE から `offline_bank.json` を直接読む（パッケージに同梱）。ONLINE は諦めるか、別途「ローカルで動く Python スクリプト」に任せる。
- **UE 内で API 直接呼び出し**: C++ で HTTP を実装し OpenAI/Gemini を呼ぶ。プロンプト・評価の移植コストが大きい。

## 4. 3D ゲームデザイン

### 4.1 イメージ

- **空間**: 1本の「廊下」。プレイヤーは正面（奥）を見て、手前に壁が迫ってくる。
- **プレイヤー**: キャラまたはマーカーが廊下の床の上。左右移動のみ（2D の左右を 3D の X 軸に相当させる）。
- **壁**: 天井から（または奥から）手前に向かって迫る「障害壁」。高さ・厚さは演出で調整。
- **ドア**: 壁に 2 つ（左・右）。正解ドアに入ればクリア、不正解ドア or 壁本体に触れればゲームオーバー。
- **問題表示**: 画面上部 or 3D 空間内の UI（問題文 + 左/右の選択肢ラベル）。既存の「左=選択肢1、右=選択肢2」を維持。

### 4.2 判定の対応

| 2D (Pygame) | 3D (UE) |
|--------------|---------|
| プレイヤー Rect と ドア Rect の重なり | プレイヤーとドアのコリジョン（Overlap / ボリューム） |
| 壁に触れた = ゲームオーバー | 壁のコリジョンに触れた = ゲームオーバー |
| 壁の落下速度 | 壁を手前に移動させる速度（難易度・問題に応じて変数で制御） |

### 4.3 カメラ

- 第三人称 or 背面追従: プレイヤー後方から廊下を見る。
- またはシンプルに正面固定: 廊下の奥を向き、壁が手前に迫る構図。  
→ まずは「固定 or 簡単な追従」で実装し、あとからカメラワークを強化。

## 5. 技術選定

| 項目 | 推奨 |
|------|------|
| エンジン | **Unreal Engine 5**（UE4 でも可。プロジェクト新規なら UE5 推奨） |

## 6. データ・API 設計（クイズサーバー）

### 6.1 問題 1 件の JSON（現行互換）

```json
{
  "q": "問題文",
  "c": ["選択肢1", "選択肢2"],
  "a": 0,
  "e": "解説",
  "src": "OpenAI|Gemini|OFFLINE"
}
```

- `a`: 正解のインデックス (0 or 1)。0=左ドア、1=右ドア。
- 画像付きは `image` / `choice_images` など拡張。第一期ではテキストのみでも可。

### 6.2 想定 API エンドポイント（例）

| メソッド | パス | 用途 |
|----------|------|------|
| POST | `/quiz/request` | 問題を 1 件 or N 件取得。body: `{ "player_id", "subject", "grade", "difficulty", "mode": "ten" \| "endless", "count" }` |
| GET  | `/quiz/offline` | オフライン用: 教科・学年で問題一覧 or 1 件取得（クエリで指定） |
| POST | `/quiz/rate` | いいね/いまいちの送信（将来） |

- 実装時は既存の `_base_prompt`・`offline_pick`・バッファ管理をサーバー側に寄せる。

## 7. フェーズ別実施計画

### Phase 0: 準備（1–2 週間）

- [x] **リポジトリ / フォルダ構成**
  - UE プロジェクト用フォルダ（例: `AIQuizUE/`）は Phase 1 で用意。既存 `AIQUIZPY避難所` に `server/` を追加済み。
- [x] **クイズ API サーバー（Python）**
  - FastAPI で `server/` を新規作成。`quiz_engine.py` でオフラインバンク読み込み・出題を実装。
  - `GET /quiz/offline` と `POST /quiz/request` を実装。環境変数は `.env.example` で記載。
- [x] **offline_bank.json の共有**
  - サーバーがプロジェクトルートの `offline_bank.json` を参照して配信。Phase 0 は「サーバーがオフライン問題を返す」で完了。

**成果物**: ローカルで動くクイズ API（Postman 等で動作確認可能）。→ **完了**（`server/README.md` 参照）。

---

### Phase 1: UE 最小プレイ可能版（2–3 週間）

- [ ] **UE プロジェクト作成**
  - テンプレート: Third Person または Blank。日本語 UI 用フォント・ローカライズは後回し可。
- [ ] **レベル**
  - 廊下（床・天井・左右壁）のシンプルな BSP または Static Mesh。終端に「壁」用のオブジェクトを 1 つ配置。
- [ ] **壁の挙動**
  - Blueprint: 壁 Actor を手前（プレイヤー方向）に一定速度で移動。速度は変数で変更可能に。
- [ ] **ドア 2 つ**
  - 壁に 2 つのドア（左・右）を配置。それぞれコリジョン付き。タグまたは変数で「左=0 / 右=1」を識別。
- [ ] **プレイヤー**
  - 既存 Third Person のキャラを使うか、簡易の Capsule + Mesh。左右移動のみ（A/D または ←/→）。前後移動は固定 or なし。
- [ ] **当たり判定**
  - プレイヤーが「壁本体」に触れた → ゲームオーバー（画面表示のみでよい）。
  - プレイヤーが「左ドア」「右ドア」に触れた → 選択 0 / 1 としてサーバーに送る代わりに、一旦ローカルで正解判定（後述）。
- [ ] **問題の取得と表示**
  - UE からクイズ API の `/quiz/request` または `/quiz/offline` を HTTP で呼ぶ。
  - 返ってきた 1 件の `q`, `c[0]`, `c[1]`, `a` を使う。
  - UMG で「問題文」「左の選択肢」「右の選択肢」を表示。正解は `a` で判定。
- [ ] **1 問のループ**
  - 問題取得 → 表示 → 壁を動かす → ドア or 壁に接触で判定 → 正解なら「正解！」表示して次の問題取得、不正解なら「ゲームオーバー」表示。
- [ ] **速度設定**
  - 壁の移動速度を難易度 or 設定に応じて変更（スライダー or 簡単/普通/難しいの 3 段階）。

**成果物**: 3D 廊下で 1 問ずつ、正解/不正解で次へ進む or ゲームオーバーになるプレイ可能ビルド。

---

### Phase 2: ゲームモード・メニュー（1–2 週間）

- [ ] **タイトル画面（UMG）**
  - 「10問チャレンジ」「エンドレス」の 2 ボタン。クリックでモードを保持して次の画面へ。
- [ ] **設定画面（またはタイトル内）**
  - 学年（1–6）、教科（算数/理科/国語）、難易度（簡単/普通/難しい）、速度（ゆっくり/標準/速い）、出題方式（ONLINE / OFFLINE）を選択可能に。  
  - 出題方式は API が ONLINE か、オフライン専用エンドポイントかを切り替えるイメージ。
- [ ] **10問チャレンジ**
  - 開始時に API で 10 問まとめて取得（既存の `fixed_questions` 相当）。順番に 1 問ずつ表示・判定。10 問正解でクリア、1 問でも不正解でゲームオーバー。
- [ ] **エンドレス**
  - 1 問ずつ API で取得（または 2–3 問バッファで先読み）。不正解まで継続。スコアは「正解数」。
- [ ] **リザルト画面**
  - スコア表示。「タイトルへ」「もう一度遊ぶ」ボタン。

**成果物**: 現行に近いフローの「タイトル → 設定 → プレイ → リザルト」が一通り動く状態。

---

### Phase 3: 演出・UX の向上（1–2 週間）

- [ ] **正解/不正解の演出**
  - 正解: エフェクト・SE・短いアニメ。不正解: ゲームオーバー用エフェクト・SE。
- [ ] **壁の速度を問題に連動**
  - 現行の `_compute_wall_speed_ratio_for_quiz` に相当するロジックをサーバー側で実装し、API で「推奨速度」を返すか、クライアントで難易度から算出。
- [ ] **解説の表示**
  - ゲームオーバー時に、問題文・正解・解説（`e`）を UMG で表示。
- [ ] **日本語フォント・テキスト**
  - フォントアセットの導入。UMG のテキストで日本語が崩れないようにする。
- [ ] **オプション: 診断モード**
  - 10問モード開始前に、算数・理科・国語の診断問題を数問出し、結果で難易度を自動設定。サーバーに診断用エンドポイントを追加するか、オフライン問題で固定するか検討。

**成果物**: 遊び心地と見た目が現行に近い水準になること。

### Phase 4: 2 人プレイ・履歴・評価（将来）

- [ ] **2 人プレイ**
  - 画面分割 or 1 画面で 2 キャラ。それぞれ別設定で問題を取得（player_id=1, 2）。同時プレイ or ターン制はデザイン次第。
- [ ] **履歴**
  - プレイ結果をローカル保存 or サーバーに送信。履歴一覧画面を UMG で実装。
- [ ] **問題評価**
  - 履歴から「いいね」「いまいち」を送信。API の `/quiz/rate` を呼び、サーバーで `quiz_ratings` に反映。

## 9. リスクと対策

| リスク | 対策 |
|--------|------|
| API キー・ネットワークを UE に持たせたくない | 必ず Python サーバーを経由させる。キーはサーバーの .env のみ。 |
| 日本語表示が乱れる | UE の Slate/UMG で日本語フォントを明示指定。フォントアセットをプロジェクトに含める。 |
| オフラインで遊びたい | サーバー未起動時は `/quiz/offline` の代わりに、UE に同梱した `offline_bank.json` を読むフォールバックを実装。 |
| 開発工数が大きい | Phase 1 を「オフラインのみ・1 問だけ」に縮小し、まず 3D の当たり判定と問題表示だけ完成させる。 |
