# Phase 1 全体像 — UE 最小プレイ可能版

**目標**: 3D 廊下で「壁が迫る → 左右に動いて正解ドアに入る」が 1 問ずつプレイでき、正解で次問・不正解でゲームオーバーになる**最小の遊べるビルド**を作る。

**想定期間**: 2〜3 週間  
**前提**: Phase 0 のクイズ API サーバーが起動し、`/quiz/request` で問題が取得できること。

- **実装手順（ステップ順）**: [UE3D_Phase1_実装手順.md](UE3D_Phase1_実装手順.md)
- **API 用 C++ コード**: リポジトリの `UnrealSource/` を UE プロジェクトにコピーして使用（[UnrealSource/README.md](../UnrealSource/README.md) 参照）。
- **プロジェクトの場所**: 推奨は「AIQUIZPY避難所の外」（例: デスクトップに `AIQuizUE` フォルダ）。詳しくは実装手順の Step 0 を参照。

---

## 1. 完成時のイメージ

- プレイヤーは**廊下の上**に立っている（奥を向く）。
- **壁**が奥側から手前（プレイヤー方向）に迫ってくる。壁には**左ドア**と**右ドア**が 2 つある。
- 画面上部（または 3D 空間内）に**問題文**と**左・右の選択肢**が表示される。
- プレイヤーは**左右移動のみ**（A/D または ←/→）で、**正解のドア**に進入するか、**不正解のドア or 壁本体**に触れるかを選ぶ。
- **正解ドア**に入る → 「正解！」表示 → 次の問題を API で取得 → 壁をリセットして再開。
- **不正解ドア or 壁**に触れる → ゲームオーバー表示（解説は Phase 3 でよい）。

つまり「**1 問のループ**」が 3D で成立している状態が Phase 1 の成果物。

## 2. やること一覧（作業の流れ）

| # | 項目 | 内容 |
|---|------|------|
| 1 | **UE プロジェクト作成** | UE5 で新規プロジェクト（Third Person または Blank）。Blueprint 中心でよい。 |
| 2 | **レベル（廊下）** | 1 本の廊下を作る。床・天井・左右の壁。BSP または Static Mesh。プレイヤーが立つ「手前」と、壁が現れる「奥」を決める。 |
| 3 | **壁 Actor** | 廊下の奥に「障害壁」を 1 枚配置。厚み・高さは任意。**2 つのドア**（左・右）を壁に開ける（穴 or 別メッシュ）。 |
| 4 | **壁の挙動** | Blueprint で「壁を手前方向に一定速度で移動」する処理。速度は変数（後で難易度連動可能）。壁がプレイヤーを通り過ぎたらリセット or 次の問題で再配置。 |
| 5 | **ドアのコリジョン** | 左ドア・右ドアそれぞれにコリジョン（Box 等）を付ける。タグまたは変数で **左=0 / 右=1** を識別。壁本体（ドア以外）にもコリジョン。 |
| 6 | **プレイヤー** | Third Person のキャラを使うか、Capsule + 簡易 Mesh。**左右移動のみ**（前後は固定 or 無効）。入力: キーボード A/D または ←/→、必要ならコントローラー左スティック。 |
| 7 | **当たり判定** | プレイヤーが「壁本体」に触れた → **ゲームオーバー**。プレイヤーが「左ドア」に触れた → 選択 **0**。プレイヤーが「右ドア」に触れた → 選択 **1**。正解インデックス `a` と比較して正解/不正解を判定。 |
| 8 | **問題の取得** | UE から Phase 0 の API に **HTTP で POST /quiz/request**（または GET /quiz/offline）。body: `subject`, `grade`, `mode: "endless"`, `count: 1` など。返ってきた **1 件**の `q`, `c[0]`, `c[1]`, `a` を使う。 |
| 9 | **問題の表示（UMG）** | Widget Blueprint で「問題文」「左の選択肢」「右の選択肢」を表示。取得した `q`, `c[0]`, `c[1]` をバインド。 |
| 10 | **1 問のループ** | ゲーム開始 or 正解後: API で 1 問取得 → UMG に表示 → 壁を奥から手前に移動開始 → プレイヤーがドア/壁に接触 → 判定 → 正解なら「正解！」表示して (10) に戻る、不正解/壁ならゲームオーバー表示。 |
| 11 | **速度設定（任意）** | 壁の移動速度を変数で切り替え（スライダー or 簡単/普通/難しい）。Phase 1 では固定値でも可。 |

## 3. データの流れ

```
[ゲーム開始 or 正解後]
    ↓
UE → HTTP POST /quiz/request (subject, grade, mode=endless)
    ↓
API → JSON 1件: { "q", "c": ["左", "右"], "a": 0 or 1, "e", "src" }
    ↓
UE: q を問題文に、c[0] を左ドア側、c[1] を右ドア側に表示
    正解インデックス a を保持（プレイヤーには見せない）
    ↓
壁を動かす → プレイヤーが左ドアに触れた → choice = 0
             プレイヤーが右ドアに触れた → choice = 1
             プレイヤーが壁に触れた → ゲームオーバー
    ↓
choice == a → 正解 → 次の問題へ
choice != a or 壁接触 → ゲームオーバー
```

## 4. UE 側で持っておくもの

- **API のベース URL**: 例 `http://127.0.0.1:8000`（エディタ実行時）。ビルド時は設定可能なように変数で持つ。
- **現在の問題データ**: `q`, `c[0]`, `c[1]`, `a`（と必要なら `e`）。
- **ゲーム状態**: 例「待機中 / 壁移動中 / 正解表示中 / ゲームオーバー」。

HTTP は Blueprint の **HTTP Request** ノード、または C++ の **Http Module** で実装。JSON のパースは **Json Blueprint Utilities** やプラグイン、または C++ で行う。

## 5. 判定まわり（2D との対応）

| 2D (Pygame) | Phase 1 (UE) |
|-------------|--------------|
| プレイヤー Rect と ドア Rect の重なり | プレイヤーとドアの **Overlap / Hit イベント**（コリジョン） |
| 壁に触れた = ゲームオーバー | 壁（ドア以外）のコリジョンに触れた = ゲームオーバー |
| 壁の落下速度（Y 増加） | 壁を手前方向に移動する **速度変数**（Tick で加算） |
| 左ドア = 選択 0、右ドア = 選択 1 | ドア Actor に **Door Index 0 / 1** を保持して渡す |

## 8. 成果物の定義

- **エディタでプレイ可能**: 廊下・壁・ドア・プレイヤー・UMG が表示され、API から 1 問取って表示し、左右移動でドアに触れて正解/不正解が判定できる。
- **ビルド可能**: パッケージしてスタンドアロンで起動し、同じフローが再現できる（API の URL は設定 or ローカルサーバー想定）。

ここまでできれば Phase 1 完了とし、Phase 2 でメニュー・10問/エンドレス・リザルトを追加する。
